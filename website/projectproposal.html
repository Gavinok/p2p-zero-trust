Term Project Proposal

The Problem

        Companies and users alike tend to assume their networks are secure end-to-end, and should it be compromised, they would know. Unfortunately, this simply isn’t the case in the modern world of cyber-security threats. These threats could be as simple as a malicious party gaining personal information sent over an insecure network, or as crippling as a total internal network breach. Such is the case when Czech cyber-security company Avast had their internal network compromised as a result of a malicious party gaining access to VPN credentials from one of their employees in 2019 [1]. To protect one’s network is to assume it has already been compromised, and to build it accordingly. One implementation of this ideology is called a Zero Trust Architecture, or a Zero Trust Network.

Zero Trust Networks Today

A popular means of securing a corporate network in 2022 is to implement an internal VPN. As is the case with Avast, sometimes this isn’t enough to stop a breach of sensitive information, since the VPN still relies on password authentication which can be taken advantage of by bad actors. Research has been done to identify what defines a Zero Trust Network, and some implementations have surfaced demonstrating the flow of true zero trust communication. A notable piece of research that will serve as a guide for us in this project is the book “Zero Trust Networks: Building Secure Systems in Untrusted Networks” by Evan Gilman and Doug Barth [2]. This book outlines important components of Zero Trust Network implementation, but falls short on outlining performance optimization, and what features can or should be omitted from a Zero Trust Network in favour of improving network performance. For this project, we want to analyse and investigate features of a Zero Trust Network that may negatively affect network performance, and decide if they can or should be omitted. If, in the future, Zero Trust Networks comprise the majority of the connections companies and users have with the rest of the internet, performance and reliability will be critical aspects to consider as usage of these networks increases.

Our Approach

The purpose of this project will be to find a middle ground providing a minimally trusting system while attempting to provide as close to 1:1 throughput and latency to a fully trusting internal network. Under the section “Realizing A Zero Trust Network” [2], Gilman and Barth outline the following requirements to create a zero trust network. This was the basis for our approach to creating a secure network.

  • All network flow MUST be authenticated before being processed
  • All network flow SHOULD be encrypted before being transmitted
  • Authentication and encryption MUST be performed by the the application-layer endpoints
  • All network flow MUST be enumerated so that access can be enforced by the system
  • The strength of authentication and encryption suites available SHOULD be used within the network
  • Authentication SHOULD not rely on public PKI providers-private PKI systems should be used instead
  • Devices SHOULD be regularly scanned, patched, and rotated

Our approach to this problem will be to fully incorporate all the required features (denoted by MUST ), while manipulating the suggested criteria in order to improve performance over a network. We will develop a “Trust Score” scaling system to ensure that as we experiment with the addition and removal of the recommended features, the network stays secure. The result we expect is a minimally trusting network with improved performance compared to a fully Zero Trust Network.
